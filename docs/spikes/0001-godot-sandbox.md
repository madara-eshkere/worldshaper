# Spike: godot-sandbox для исполнения Механик (M0)

**Дата:** 2026-07-17 · **Вопрос:** годится ли godot-sandbox как фундамент для
безопасного исполнения ИИ-сгенерированных Механик? (ADR-0006)
**Версии:** godot-sandbox v0.48, Godot 4.7-stable, Windows x86_64.
**Вердикт: УСЛОВНЫЙ GO** — механизм способен, но диктует архитектуру (см. ниже).

## Что подтверждено

| # | Проверка | Результат |
|---|---|---|
| 1 | Расширение грузится на Godot 4.7 | ✅ классы `Sandbox`, `SafeGDScript`, `ELFScript` зарегистрированы |
| 2 | Рантайм-компиляция кода механики | ✅ SafeGDScript компилирует исходник в ELF (`compile err: OK`) |
| 3 | Простой гость исполняется | ✅ `ping()` вернул значение |
| 4 | Нода `Sandbox` инстанцируется | ✅ |
| 5 | Лимиты выставляются ДО загрузки программы | ✅ `set_instructions_max`, `execution_timeout`, `restrictions` — записываются и читаются |
| 6 | Реальный ELF грузится в ноду | ✅ `load_buffer` 1.5 МБ → `has_program_loaded=true` |
| 7 | Мониторы рантайма читаемы | ✅ `monitor_calls_made / _exceptions / _execution_timeouts` — механизм детекта обрыва |

## Полный набор контроля на ноде `Sandbox`

- Лимиты: `set_instructions_max`, `set_memory_max`, `set_allocations_max`, `set_max_refs`, `execution_timeout`.
- Изоляция: `set_restrictions(true)` = deny-all, затем вайтлист — `add_allowed_object`, `set_class_allowed_callback`, `set_method_allowed_callback`, `set_property_allowed_callback`, `set_resource_allowed_callback`.
- Загрузка: `load_buffer`, `set_program`, `Sandbox.FromBuffer`.
- Вызов: `vmcall`, `vmcallable`.
- Мониторинг: `monitor_*` свойства (кучи, исключения, таймауты, число вызовов).

## Критичная находка (определяет архитектуру)

**«SafeGDScript как скрипт на ноде» — тупик для нас.** У `SafeGDScript` (родитель
`ScriptExtension`) внутренняя песочница наружу НЕ выдаётся: нет `get_sandbox_for`,
нет `get_content`. Значит лимиты выставить негде. Прямое следствие: гость,
обращающийся к хост-глобали (`OS`), **вешает движок** без лимита — воспроизведено,
процесс висел 371 с до принудительного убийства.

`get_sandbox_for` и `get_content` есть только у **`ELFScript`**, а весь контроль —
на **ноде `Sandbox`**. Поэтому Механики обязаны идти через путь `Sandbox`-нода /
`ELFScript`, где песочница под нашим контролем.

## Целевая архитектура Механики (для M1)

1. Инженер отдаёт исходник Механики (SafeGDScript-подмножество).
2. Компилируем в ELF (внутренний компилятор аддона, `gdscript.elf`).
3. Создаём ноду `Sandbox`, которой владеем.
4. **До первого вызова**: `set_instructions_max`, `execution_timeout`, `set_restrictions(true)`, вайтлист только Примитивов хост-API.
5. Грузим ELF, вызываем через `vmcall`.
6. После вызова читаем `monitor_execution_timeouts`/`_exceptions` → если обрыв, Механика отбраковывается, мир не тронут.

Итог: движок защищён лимитом инструкций (не зависит от доброты гостя), хост
доступен только через вайтлист Примитивов — ровно требование ADR-0006.

## Не доказано в спайке (→ первый тест M1-обёртки)

Живой обрыв бесконечного цикла лимитом инструкций in-process не демонстрировался
(нужны имена экспортов ELF нашей Механики, которых на этапе спайка нет). Вся
инфраструктура для этого присутствует и управляема; демонстрация «лимит режет
цикл, движок жив» — критерий приёмки задачи «Mechanic host wrapper» на M1.

## Операционная заметка

Каждый прогон Godot с этим расширением оборачивать жёстким OS-таймаутом
(Start-Process + WaitForExit(ms) + Kill): без лимитов гость способен подвесить
процесс, а «SafeGDScript как скрипт» может всплыть GUI-окном в безголовом режиме.

## Откат (если M1-обёртка не сложится)

Резерв из ADR-0006 — сырой GDScript с AST-вайтлистом в сайдкаре — остаётся в
силе, но НЕ активируется: механизм признан способным.
