## Закрытые (M1 майлстоун-багтест, раунд 3 — перепил боевки/ходов, 2026-07-20)

### B-024 — Пошаговая система была построена ВОКРУГ игрока (корень многих багов)
- **Причина:** ходы шли «игрок сходил → все мобы по разу до/после». Мобы не были
  независимы, скорость не давала лишних ходов → при побеге зазор до быстрого моба
  оставался постоянным; фазы «до/после» плодили «один бьёт дважды, другой раз»;
  тайминг фаз пускал игрока и моба в одну клетку.
- **Решение:** `turn_scheduler.gd` — энергетический планировщик (как в роглайках):
  у каждого актёра своё время следующего хода `next_at`, ход получает тот, чьё время
  подошло; после хода `next_at += 1/speed`. Быстрый (speed 8) реально ходит ~вдвое
  чаще speed-4 и догоняет. Игрок — один из актёров, планировщик крутит мобов до его
  хода. `enemy_controller` удалён, его AI переехал в планировщик. Тест `test_speed`
  доказывает: за N ходов игрока быстрый моб сближается сильнее медленного.

### B-025 — Движение неотзывчивое / тапы «съедались»
- **Причина:** окно диагонали (0.06с) + кулдаун в `_process` откладывали и глотали
  быстрые нажатия (клавиша отпущена до коммита окна → ход терялся).
- **Решение:** действие на нажатие сразу, с коалесценцией одного кадра (ловит
  диагональ), авто-повтор при зажатии. Тапы не теряются.

### B-026 — Двойная атака / коллизия в одну клетку / `player_attacked` после `died`
- **Причина:** опрос движения бил каждый кадр (см. B-022) + фазы. Плюс проверка
  занятости клетки была нестрогой.
- **Решение:** планировщик даёт по одному действию за ход; `_can_enter` строго
  запрещает клетку игрока и любого `blocking`-актёра. События переименованы (ниже).

### B-027 — Имена событий `attack`/`died` неоднозначны
- **Решение:** однозначные события: `player_hit_enemy`, `enemy_hit_player`,
  `enemy_slain {name}`, `player_died`. Мок-ГМ даёт персональную реплику по `name`
  (гоблин/мышь — разные).

### B-028 — Мобы одинаковые (вид/реплики); летучая мышь падала в яму
- **Решение:** классы существ данными (`bestiary.gd`): hp/atk/def/скорость/полёт/цвет/
  форма. Гоблин — красный круг, мышь — фиолетовый треугольник, летает (`flying`) и
  НЕ падает в ямы (планировщик пропускает яму для летающих). Реплики персональные.

## Закрытые (M1 майлстоун-багтест, раунд 2, 2026-07-20)

### B-022 — Бой спамил атаки / игра «слишком быстрая» (привязана к кадрам)
- **Причина:** движение опрашивалось в `_process` каждый кадр, но гейт был только
  твин движения; атака/удар об объект твин не создают → одно зажатие клавиши к
  врагу било его КАЖДЫЙ кадр (лог `attack,player_attacked,attack,attack,died,...`),
  враг контратаковал каждый кадр, а после смерти врага-трупа герой в тот же зажим
  шагал на него.
- **Решение:** введён кулдаун действия `ACT_REPEAT_SEC` (0.14с), гейтящий любое
  зажатое движение/атаку. Одно нажатие = одно действие; зажатие = ровный темп.
  Тест скорости покрывает раунды. Заодно ушло ощущение «слишком быстро».

### B-023 — Диагональ требовала идеально одновременного нажатия и стоила 2 хода
- **Причина:** движение шло по событию клавиши; W затем D = два кардинальных шага
  (2 хода), «диагональ» ловилась только при точном совпадении кадра.
- **Решение:** движение переведено на опрос удержанных клавиш + окно `DIAG_WINDOW`
  (0.06с) на первом нажатии — две клавиши в окне читаются как один диагональный шаг
  = один ход (как у NPC).

### Механика скорости (новое, по запросу — не хардкодить очередь хода)
- `speed` из DEX (`Speed.for_dex`), пороги крупные; `move_cells` = клетки за ход
  (≤6→1, 7-9→2, 10→3). Очередь: враги быстрее игрока ходят на `player_turn_started`
  (до игрока), медленнее — на `player_turn_ended`. Гоблин медленнее игрока, летучая
  мышь (speed 8) — быстрее и ходит 2 клетки. Показано в Листе/статус-HUD. Балансить
  пороги позже.

## Закрытые (M1 майлстоун-багтест, раунд 1, 2026-07-20)

### B-006 — Яма рисовалась не в той клетке и «не работала»
- **Причина:** геометрия уровня (размеры, клетка ямы, стены) была захардкожена в
  `world_grid.gd` (движковое представление) И продублирована в `level.gd` с другими
  координатами — рассинхрон. Симптом полной архитектурной проблемы.
- **Решение:** карта вынесена в World State как данные (тайлы), `world_grid` стал
  чистым рендерером, `level.gd` авторит карту. Яма рисуется из своего World-объекта
  (`props.revealed`) в `entity_renderer`. Дублировать нечего. Ревизия: CODE-REVIEW-2026-07-20.md.

### B-005 — Реплика нарратора мгновенно исчезала
- **Причина:** очередь реплик в `narrator_ui` плодила перекрывающиеся твины; новый
  наследовал фазу затухания старого.
- **Решение:** переписан на «одна активная реплика»: новая убивает старый твин
  (`_tween.kill()`) и стартует чистый reveal→hold→fade.

### B-008 / B-015 — Диагональ уродская и стоила 2 хода
- **Причина:** движение шло по событию нажатия; W затем D давали два отдельных
  шага (два хода), «диагональ» была фикцией с окном тайминга.
- **Решение:** движение переведено на опрос удержанных клавиш в `_process`; две
  клавиши одновременно = один диагональный шаг = один ход (как у NPC). Каденс задаёт
  твин движения.

### B-007 — Враг не проваливался в яму
- **Причина:** `enemy_controller._step_toward` пропускал любую занятую клетку
  (`objects_at(cand).is_empty()`), а яма — объект.
- **Решение:** `_can_enter` пускает на клетку, если там нет `blocking`-объекта и не
  игрок (яма не `blocking`); `_check_hazards` при входе на яму ставит врагу стан.

### B-012 / B-016 — После смерти игра продолжалась; труп добивали и блокировал проход
- **Причина:** не было состояния game-over; мёртвый враг сохранял теги
  `enemy`/`blocking`, поэтому bump-to-attack бил его снова (спам died/narration),
  а тег `blocking` не давал пройти.
- **Решение:** на `player_died` → `get_tree().paused = true` (весь tree замирает
  разом, без per-entity freeze) + оверлей + R-рестарт. `combat.gd` при смерти врага
  снимает `enemy`/`blocking`, вешает `corpse`/`destructible` — труп не бьётся как
  враг и не блокирует.

### B-013 — Победа не заканчивала игру
- **Причина:** `level_complete` только эмитился, состояния конца не было.
- **Решение:** `game_over_ui` на `level_complete` тоже паузит tree + оверлей «Уровень пройден» + R.

### B-014 — Зелье лечило выше максимума HP
- **Причина:** `Primitives.heal` делал `hp += amount` без потолка.
- **Решение:** `heal` клампит по `hp_max`, если он есть у объекта. Тест в test_combat (8+5→10).

### B-017 — Кадр конца игры на клетку раньше
- **Причина:** на `level_complete` tree паузится, пока твин движения игрока ещё
  скользит → игрок замирал на прежней позиции.
- **Решение:** `player._on_game_event` на конце игры убивает твин и снапит позицию
  на текущую клетку до паузы.

### B-018 — Реплика смерти/победы играла только в консоли после рестарта
- **Причина:** пауза tree останавливала `SidecarClient` (polling) и `narrator_ui`
  (твин) → реплика не доходила/не рисовалась.
- **Решение:** оба узла переведены в `PROCESS_MODE_ALWAYS` — сайдкар продолжает
  получать реплику, нарратор показывает её поверх оверлея.

### B-009 — Не было индикатора убийства врага
- **Решение:** `player.gd` эмитит `enemy_defeated` при hp≤0 цели; у мок-ГМ есть реплика.
