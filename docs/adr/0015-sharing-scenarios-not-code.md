# Шеринг: сценарии и персонажи как текст-данные; Механики/код — не шарятся

Решение пользователя (2026-07-18) после разбора угроз. Механики — композиции
Примитивов; шарить их бессмысленно и опасно: композиция зависит от Примитивов
получателя, а сгенерированные Примитивы у него другие (или совпадают имена при
разном поведении → катастрофа). Значит:

1. **Шеринг Механик и Примитивов — НЕ строим.** Композицию из одних ванильных
   Примитивов получатель и сам регенерит Сборщиком за копейки — ценности мало,
   риска (переносимый код) много.
2. **Шеринг персонажей и сценариев — ДА**, и это ключевая социальная фича. И то, и
   другое — **чистый текст/данные**: персонаж = лор-промпт + статы; сценарий = вывод
   Режиссёра (сеттинг, история, карта, Скриптовые триггеры как данные, NPC, объекты,
   условие победы как предикат). Кода там нет — Режиссёр генерит сценарий без кода.
3. **Результат работы Инженера (Примитивы/Механики для этого сценария) НЕ входит в
   шаринг** — обезопасить переносимый код неподъёмно. Получатель импортирует
   текст-план, и **его** игра локально регенерит нужные Примитивы/Механики своим
   Инженером, прогнав через свой Валидатор.
4. **Телеметрия «к нам» — можно** (собираем Механики/Примитивы игроков для пополнения
   Ванильной библиотеки к релизу), но обратно в билд — только через НАШЕ ревью;
   автоматического вливания извне нет.

Следствия: переносимые артефакты — всегда данные, никогда исполняемый код →
поверхность атаки через шеринг закрыта по построению; получатель всегда прогоняет
импорт через свой Валидатор (проходимость, Канон); «социалка» игры (обмен
приключениями и билдами) сохраняется без риска RCE.
